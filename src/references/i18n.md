# Multilingual Sites (i18n)

Automad supports two common multilingual patterns, plus dashboard-side translations. Pick deliberately — they're not all equivalent in cost.

## `@{ :language }` — the current language variable

Every page knows its `:language`. Use it on the root element so the browser picks the right hyphenation/font/SR voice:

```html
<html lang="@{ :language | def('en') }">
```

Use it to pick between per-field translations and per-tree translations below.

## Pattern A — per-language page trees

Mirror the page tree under `pages/de/...` and `pages/en/...`. A dispatcher at the top switches language for sibling pages:

```
pages/
├── de/
│   ├── home.txt
│   └── about.txt
└── en/
    ├── home.txt
    └── about.txt
```

Pros: native URL structure (`/de/kontakt` vs `/en/contact`), one content tree per language, SEO-friendly (hreflang support). Cons: pages must be created twice; renaming a page means editing both trees.

### Language switcher pattern

Build a list of language roots and link to the page's sibling in the other language:

```html
<nav class="lang">
  <@ newPagelist { type: 'children' } @>
  <@~ foreach in pagelist ~@>
    <a href="@{ :url }" <@ if @{ :current } @>class="active"<@ end @>>@{ :title }</a>
  <@~ end ~@>
</nav>
```

Read the `:url` and `:title` of each language root as the menu items.

## Pattern B — per-field translations

Each translatable field has both `*_de` and `*_en` siblings; the template picks by `:language`:

```html
<@ if @{ :language } == 'de' @>
  <h1>@{ textHeroTitle_de | def('@{ :title }') }</h1>
<@ else @>
  <h1>@{ textHeroTitle_en | def('@{ :title }') }</h1>
<@ end @>
```

Or with a shared def helper:

```
<@~ set { t: '@{ language }' } ~@>
<h1>@{ ( 'textHeroTitle_' + t ) | def('@{ :title }') }</h1>
```

(Note: concatenation like the above requires `<@ def "..." @>` if you need to build the field name dynamically — see `template-syntax.md`. When in doubt, use the explicit `if/else` form; it's less clever and easier to maintain.)

Pros: one page per URL slot, everything else stays in sync. Cons: many fields in `theme.json` (`textHeroTitle_de`, `textHeroTitle_en`, `textHeroLead_de`, `textHeroLead_en`, …); tooltips must repeat per language.

This is the pattern many small two-language sites fall back to.

## Pattern C — mixed (recommended for two-language sites)

Combine:
- Per-tree for **top-level navigation and SEO** (page titles, slugs, URL structure)
- Per-field for **page content strings** (CTAs, button labels, hero copy)

The existing `goldener-hirsch` theme is an example of a DE/EN hotel site using this kind of hybrid setup.

## Locale and date formatting

The standard theme defines a `locale` field (per-page or shared) for date formatting:

```
@{ date | dateFormat(@{ formatDatePost | def('l, F jS Y') }, @{ locale }) }
```

This way `1. April 2025` (de_DE) vs `April 1, 2025` (en_US) come from the same data. Set `locale` in shared data once per language tree.

`@{ locale }` defaults to the page's `:language` if you don't override it.

## Dashboard translations — `i18n/*.json`

Automad loads dashboard label translations from `i18n/<locale>.json` in your theme folder:

```
my-theme/i18n/
├── de.json
└── en.json
```

These translate dashboard-side strings ("Apply and Reload", "Block Editor", "Page Settings") into the editor's UI when the user is editing that language. They do NOT translate the page content — only Automad's chrome.

Format (an excerpt from a typical translation file):

```json
{
  "btnSave": "Speichern",
  "btnApply": "Übernehmen und neu laden",
  "tabBlocks": "Block-Editor"
}
```

Field-level tooltips from `theme.json → tooltips.*` are NOT auto-translated. To support translated tooltips, list both languages in `theme.json`:

```json
"tooltips": {
  "+main_de": "Hauptinhalt",
  "+main_en": "Main content"
}
```

Then the dashboard shows the right tooltip based on `:language`. (Verify against the running dashboard — exact behavior depends on Automad version.)

## Things to NOT do

- **Don't combine per-tree and per-field URLs** — pick one structure for URLs. Mixing `pages/de/index.txt` with `pages/contact_de.txt` is a maintenance nightmare.
- **Don't rely on `:language` to drive business logic** — it's a hint, not enforced. Validate upstream if needed.
- **Don't forget `lang="..."` on `<html>`** — it changes how text breaks and reads aloud; omitting it is an accessibility miss.
